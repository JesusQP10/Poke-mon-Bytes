1. Introducción al Proyecto

Pokémon Bytes es una aplicación web de rol y aventura por turnos inspirada en la segunda generación de la franquicia Pokémon (Pokémon Oro y Plata, Nintendo, 1999). El proyecto recrea en el navegador la experiencia de explorar la región de Johto: elegir un Pokémon inicial, caminar por mapas en 2D con estética Game Boy Color, encontrarse con Pokémon salvajes, combatir por turnos y gestionar un equipo de hasta seis criaturas.

La aplicación se divide en dos grandes bloques: un backend REST desarrollado en Java con Spring Boot 3, que gestiona toda la lógica de negocio, la persistencia en MySQL y la seguridad mediante JWT; y un frontend SPA desarrollado con React 19 y el motor de videojuegos Phaser 3, que renderiza el mundo del juego, los mapas exportados desde Tiled Map Editor, y las pantallas de combate y menús.

El nombre Bytes es un guiño consciente a la naturaleza digital del proyecto, fusionando el universo Pokémon con la terminología informática. Este proyecto integra los conocimientos de todas las asignaturas del ciclo formativo DAW: bases de datos relacionales, APIs REST, seguridad web, desarrollo frontend reactivo y motor de videojuegos en tiempo real.

3. Casos de Uso

Autenticación

  • CU-01 Registrarse: el visitante envía username y password; el servidor crea la cuenta con hash BCrypt y devuelve 201 Created.
  • CU-02 Iniciar sesión: el servidor valida credenciales con AuthenticationManager y devuelve un token JWT firmado HS256, válido 24 horas.

Gestión de partida

  • CU-03 Crear nueva partida: el jugador introduce nombre (máx. 10 caracteres; predefinidos: Jesús, Oro, Ethan) y elige uno de los tres iniciales de Johto (Chikorita 152, Cyndaquil 155, Totodile 158).
  • CU-04 Continuar partida: se carga el estado desde MySQL (GET /api/v1/juego/estado); si no hay red, desde localStorage (pokemon_bytes_save_v1).
  • CU-05 Guardar partida: persiste posición, mapa y JSON de estado en MySQL; duplica localmente en localStorage como caché de emergencia.
  • CU-06 Reiniciar partida: borra equipo, inventario y PP en BD; restablece dinero a 300 y posición a player-room (5,7).

Exploración y Combate

  • CU-07 Moverse por el mapa: movimiento tile a tile con WASD/Flechas; colisión verificada antes de cada paso.
  • CU-08 Interactuar con NPC: la tecla Z activa el sistema de radar, busca el interactuable más cercano y dispara el diálogo estilo GBC.
  • CU-09 Cambiar de mapa: los warps de Tiled disparan WarpSystem con transición fade de 250 ms.
  • CU-10 Encuentro salvaje: al caminar en hierba alta, SistemaEncuentros genera un encuentro tras 4-7 pasos ponderados.
  • CU-11 Combatir por turnos: el jugador elige movimiento; el servidor calcula daño, estados y PP.
  • CU-12 Capturar Pokémon: el jugador lanza una Ball; el servidor ejecuta el algoritmo de captura Gen II.

Economía

  • CU-13 Comprar ítems: transacción ACID que descuenta dinero y actualiza inventario.
  • CU-14 Recibir ítems por evento: recompensas narrativas (poción del PC, ayudante) añadidas por el servidor.

8. Módulos del Backend

8.1 Módulo de Seguridad

Este módulo protege todos los endpoints de la API mediante JSON Web Tokens. Está compuesto por tres clases que trabajan conjuntamente:

ConfiguracionSeguridad.java — @Configuration @EnableWebSecurity
Define dos cadenas de seguridad con @Order:
• cadenaActuator (@Order 1): aplica solo a /actuator/**; permite cualquier petición sin JWT para que el health-check funcione sin token.
• cadenaFiltroSeguridad (@Order 2): desactiva CSRF (API stateless), configura CORS (orígenes: localhost:*, 127.0.0.1:*, 192.168.*:*, 10.*:*, 172.*:*), sesiones STATELESS, añade FiltroAutenticacionJwt antes de UsernamePasswordAuthenticationFilter, y declara rutas públicas (/auth/**, /swagger-ui/**) y privadas (anyRequest().authenticated()).
ServicioJwt.java — @Service
Genera y valida tokens JWT con JJWT 0.12.5:
• generarToken(UserDetails): crea un JWT con subject = username, issuedAt = ahora, expiration = ahora + 24 h, firmado con HMAC-SHA256 y la clave de jwt.secret.key.
• extraerNombreUsuario(token): extrae el subject del payload.
• esTokenValido(token, userDetails): comprueba que el username coincide y el token no está expirado.
FiltroAutenticacionJwt.java — @Component, extiende OncePerRequestFilter
Se ejecuta una vez por petición:
1. Lee el header Authorization: Bearer <token>. Si no existe, pasa la petición sin autenticar.
2. Extrae el username del token con ServicioJwt. Si el token está malformado o expirado, pasa sin autenticar.
3. Si el SecurityContext está vacío y el token es válido, crea un UsernamePasswordAuthenticationToken y lo establece en el contexto.

8.2 Módulo de Autenticación

ControladorAutenticacion.java — @RestController @RequestMapping("/auth")
Dos endpoints públicos:
• POST /auth/registrar: recibe {username, password} validado con @Valid, llama a ServicioAutenticacion.registrarUsuarioNuevo() y devuelve 201.
• POST /auth/iniciarSesion: autentica con AuthenticationManager, obtiene el UserDetails y genera el JWT con ServicioJwt.generarToken().
ServicioAutenticacion.java — @Service
• registrarUsuarioNuevo(Usuario): verifica unicidad del username, codifica la contraseña con BCryptPasswordEncoder y persiste el usuario.
• cargarUsuarioPorNombreUsuario(username): busca el usuario en BD para emitir el JWT tras login.
ServicioDetallesUsuario.java — @Service, implementa UserDetailsService
Adaptador entre Spring Security y el repositorio de usuarios. El método loadUserByUsername es llamado por el FiltroAutenticacionJwt en cada petición autenticada para cargar los detalles del usuario desde BD.

8.3 Módulo de Juego

JuegoController.java — @RestController @RequestMapping("/api/v1/juego")
6 endpoints, todos requieren JWT. Delega completamente en JuegoService:
GET /estado · POST /starter · GET /equipo · POST /guardar · POST /reiniciar · POST /inventario/anadir.
JuegoService.java — @Service
Lógica central de la partida. Métodos principales:

• obtenerEstado(username): @Transactional(readOnly=true). Devuelve el estado completo: equipo ordenado por posición (0-5), dinero, inventario, mapa, posición y el JSON del cliente parseado con Jackson. Si el blob JSON está corrupto, devuelve Map.of() en lugar de romper el GET.

• elegirStarter(username, starterId): @Transactional. Idempotente: si ya tiene equipo, devuelve el primer Pokémon. Starters permitidos: [152, 155, 158]. Crea un PokemonUsuario a nivel 5 con stats calculadas desde PokedexMaestra (mínimos garantizados: HP≥20, demás stats≥5). Sprite URL apunta al repositorio oficial de PokéAPI en GitHub.

• guardarPartida(username, body): @Transactional. Actualiza posX, posY, mapaActual y el JSON del cliente. El JSON puede llegar como Map (lo serializa Jackson) o como String ya serializado. No toca dinero ni inventario: esos campos se gestionan exclusivamente por sus propios endpoints para evitar trampas.

• reiniciarPartida(username): @Transactional. Borra todos los PP (estadoMovimientoRepo), borra todos los Pokémon del equipo y caja, vacía el inventario, restablece dinero=300, mapa=player-room, posX=5, posY=7, estadoClienteJson=null.

• anadirAlInventario(username, itemId, nombreItem, cantidad): @Transactional. Resuelve el ítem por ID o por nombre. Busca la fila existente en InventarioUsuario; si existe, suma la cantidad; si no, crea una nueva fila. Devuelve el inventario completo actualizado.

• toDto(PokemonUsuario): helper privado que enriquece el Pokémon con datos de PokedexMaestra (nombre, tipos normalizados a inglés, sprite URL). Devuelve un Map compatible con el frontend Phaser.

8.4 Módulo de Batalla

BatallaController.java — @RestController @RequestMapping("/api/v1/batalla")
5 endpoints. Los helpers enteroOpcional() e idLargoOpcional() toleran cuerpos null o valores Number/String para máxima compatibilidad con el cliente Phaser.
BatallaService.java — @Service
Motor de combate principal. Mantiene:
• learnsetCache: ConcurrentHashMap<Integer, List<EntradaLearnset>>: caché de movimientos por especie para no consultar PokéAPI en cada turno. Versión objetivo: gold-silver.
• cachedSalvajesUserId: volatile Long: caché del ID de la cuenta técnica de salvajes.

ejecutarTurno(username, SolicitudTurno): @Transactional
Flujo completo:
1. Carga atacante y defensor; valida que el atacante pertenece al usuario.
2. Resuelve el movimiento (por ID, legacy, o primer slot con PP).
3. Pre-turno: procesa estados (parálisis 25% falla, sueño espera, congelación 20% despierta).
4. Verifica precisión con CalculoService.verificaImpacto().
5. Si es movimiento de estado (sin potencia), aplica efectos y devuelve sin daño.
6. Calcula efectividad (TipoService), STAB y golpe crítico (×2.0).
7. CalculoService.calcularDanio() → aplica al HP del defensor.
8. Post-turno: quemadura (hpMax/8), tóxico progresivo (contadorTóxico×hpMax/16), drenadoras.
9. Guarda ambos Pokémon en BD y construye RespuestaTurno.

prepararInstanciaSalvaje(pokedexId, nivel): @Transactional
Crea un PokemonUsuario con usuarioId = cuentaSalvajes y posicionEquipo = 100. Stats: HP = statBase + 10 (mín. 20); demás: statBase (mín. 5).

liberarInstanciaSalvaje(pokemonUsuarioId): @Transactional
Verifica que el Pokémon pertenece a la cuenta técnica, borra sus PP y lo elimina de BD.
CalculoService.java — @Service
Fórmulas puras de combate (sin efectos secundarios, testables directamente):
• calcularDanio(N, A, D, P, efectividad, stab, estado, esFisico): Fórmula Gen II. Penalización quemadura físico: A/2. STAB: ×1.5. Variación: ThreadLocalRandom [0.85, 1.0). Mínimo garantizado: 1.
• verificaImpacto(precisionBase): si ≥100, siempre impacta; si no, random(1,101) ≤ precision.
• fueGolpeCritico(): probabilidad 6.25% (random() < 0.0625).
• calcularExperiencia(nivel, xpBase): Math.floor(xpBase × nivel / 7.0).
• calcularCaptura(hpMax, hpActual, captureRate, bonoBall, estado): Algoritmo Gen II simplificado. Bonus: Dormido/Congelado ×2.0; Paralizado/Quemado/Envenenado ×1.5. Si a≥255 garantizado; si no, random(0,256) < a.
TipoService.java — @Service
Consulta la tabla TIPOS en BD (no hardcoded).
• Normaliza tipos PokéAPI (inglés) y del juego (español) al nombre canónico almacenado en BD (17 entradas en NORMALIZAR).
• calcularEfectividad(tipoAtaque, def1, def2): multiplica los dos multiplicadores (tipo1 × tipo2) para defensores con doble tipo.
• mensajeEfectividad(mult): devuelve texto estilo Gen II ("¡Es muy efectivo!", "No es muy efectivo...", "¡No tiene efecto!").

8.5 Módulo de Tienda

TiendaController.java — @RestController @RequestMapping("/api/v1/tienda")
Un único endpoint: POST /comprar con SolicitudCompra {itemId, cantidad} validado con @Valid.
TiendaService.java — @Service
comprarItem(username, SolicitudCompra) — @Transactional:
1. Carga el usuario y el ítem (lanza 404 si no existen).
2. Calcula costoTotal = precio × cantidad.
3. Si el saldo es insuficiente, lanza ErrorNegocio con mensaje formateado en ₽ → la transacción revierte sin tocar BD.
4. Descuenta el dinero y guarda el usuario.
5. Busca la fila en InventarioUsuario por InventarioId(usuarioId, itemId); si no existe, crea una nueva con cantidad 0.
6. Suma la cantidad y guarda.
7. Devuelve mensaje con formato "Compraste N × Ítem. Dinero restante: X₽", saldo restante e inventario completo.

8.6 Módulo de Inicialización de Datos (Seeding)

Cuatro CommandLineRunner que se ejecutan al arrancar el servidor. Todos comprueban si la tabla ya tiene datos antes de ejecutarse, evitando re-descargas.

SembradorUsuarioSalvajes.java — @Component @Order(1)
El primero en ejecutarse (Order 1). Crea el usuario técnico con username de CuentaSalvajes.USERNAME y contraseña BCrypt de "NO_LOGIN" (sin acceso humano). Mapa: "__sistema__". Dinero: 0.
InicializadorTipos.java — registrado como @Bean en PokemonBackendApplication (sin @Component para evitar doble registro).

12. Diario de Desarrollo

Fase I – Seguridad JWT con Spring Security 6

Dificultad: Spring Security 6.x eliminó WebSecurityConfigurerAdapter. La documentación oficial mezcla ejemplos de versiones anteriores, y los tutoriales más comunes estaban desactualizados. El nuevo modelo basado en beans de SecurityFilterChain es más explícito pero requiere entender el flujo de filtros de Spring.

Decisión: Se implementaron dos SecurityFilterChain con @Order: el de Actuator (sin JWT) en orden 1 y el principal en orden 2. Esto permite que el health-check funcione siempre sin necesidad de token, útil para monitorización. El FiltroAutenticacionJwt extiende OncePerRequestFilter para garantizar que solo se ejecuta una vez por petición.

Fase II – Motor de Combate Gen II

Dificultad: La fórmula de daño de Gen II tiene múltiples modificadores que interactúan entre sí: STAB, efectividad de tipos, golpe crítico, variación aleatoria y la penalización por quemadura. Además, los nombres de tipos vienen en inglés desde PokéAPI pero la BD los almacena en español, lo que causaba que la efectividad siempre devolviera 1.0.

Decisión: Se separaron las fórmulas matemáticas puras en CalculoService (testeable sin Spring) y la lógica de estado en BatallaService. Se creó TipoService con un mapa de normalización de 30 entradas (inglés ↔ español ↔ variantes con tilde) para que la búsqueda en BD siempre encuentre el multiplicador correcto.

Fase III – Inventario con Clave Compuesta

Dificultad: Modelar la relación M:N entre Usuario e Item con campo de cantidad no es posible con colecciones JPA estándar (@ManyToMany no soporta campos extra). La primera implementación con @EmbeddedId fallaba por no implementar correctamente equals() y hashCode() en la clave.

Decisión: Lombok @Data en InventarioId genera automáticamente equals/hashCode correctos. La operación de compra en TiendaService usa findById(inventarioId); si no existe, crea una nueva fila con cantidad 0 antes de sumar, evitando null pointer.

Fase IV – Seeding Reactivo desde PokéAPI

Dificultad: La primera versión del seeding hacía peticiones secuenciales a PokéAPI, tardando más de 10 minutos en cargar 251 Pokémon y lanzando DataBufferLimitException porque el buffer por defecto de WebClient es de solo 256 KB, insuficiente para las respuestas grandes de PokéAPI.

Decisión: Se configuró el buffer a 16 MB en ServicioPokeApi con ExchangeStrategies. Se usó Flux.flatMap(..., 5) para limitar la concurrencia y .buffer(20) para inserciones en batch, reduciendo el tiempo de seeding a menos de 2 minutos.

Fase V – Integración React + Phaser 3

Dificultad principal del frontend: Phaser gestiona su propio bucle de juego completamente independiente de React. Los hooks de Zustand (useStore) no funcionan fuera del árbol de componentes React; Phaser no tiene acceso a ellos.

Decisión: PuenteApi.js usa la API imperativa de Zustand (usarJuegoStore.getState()) para leer y escribir el estado desde Phaser sin necesidad de hooks. Phaser llama a PuenteApi para toda comunicación HTTP; PuenteApi actualiza el store; React se re-renderiza reactivamente.

Sistema de Colisiones por Grid

Dificultad: El sistema de física continua de Phaser causaba que el jugador atravesara paredes en esquinas a velocidades altas y que la cámara se desincronizara al cambiar de mapa. Los tilemaps de Tiled requerían configuración precisa de capas y propiedades de colisión.

Decisión: Se implementó movimiento tile a tile: el jugador solo se desplaza si el tile destino está libre (sin colisión). El input se bloquea durante los 200 ms del movimiento para evitar superposiciones. Esto simplificó radicalmente el código y eliminó todos los bugs de colisión.

Consistencia del Sistema de Guardado

Dificultad: Al sincronizar el estado del servidor con el store local, los clientes con versiones antiguas del JSON guardaban el inventario dentro del estadoClienteJson. Al hidratar desde el servidor, el inventario se duplicaba (inventario BD + inventario JSON).

Decisión: Se estableció la regla: dinero e inventario del servidor nunca se sobreescriben con los del JSON del cliente en setPlayerState(). La función normalizarListaInventario() fusiona filas con el mismo itemId o nombre antes de guardar en el store, eliminando duplicados de cualquier origen.
