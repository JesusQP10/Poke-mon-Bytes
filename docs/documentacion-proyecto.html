1. Introducción

1.1 Descripción y Motivación

Pokémon Bytes es una aplicación web de rol y aventura por turnos inspirada en la segunda generación de la franquicia Pokémon (Pokémon Oro y Plata, 1999). El proyecto recrea la experiencia de explorar la región de Johto desde cualquier navegador web moderno, permitiendo al jugador elegir un Pokémon inicial, explorar mapas en 2D con estilo Game Boy Color, encontrarse con Pokémon salvajes, combatir por turnos y gestionar un equipo de hasta seis criaturas.

El nombre Bytes es un guiño consciente a la naturaleza digital del proyecto y a la terminología informática, fusionando el universo Pokémon con el mundo del desarrollo de software. El proyecto integra los conocimientos de todas las asignaturas del ciclo formativo DAW: diseño de bases de datos relacionales, programación de APIs REST, seguridad mediante JWT, desarrollo de interfaces reactivas con React y gestión de un motor de videojuegos en tiempo real con Phaser 3.

1.2 Problema que Resuelve

Los videojuegos de Pokémon originales funcionan exclusivamente en consolas portátiles Nintendo, lo que limita su acceso a quienes disponen del hardware específico. Pokémon Bytes democratiza esa experiencia: cualquier persona con un navegador moderno puede jugar sin instalar nada.

Técnicamente, el proyecto demuestra cómo integrar un motor de videojuegos en tiempo real (Phaser 3) dentro de una SPA React, delegando toda la lógica crítica de juego (fórmulas de daño, captura, economía) al servidor Spring Boot para garantizar integridad y prevenir trampas desde el cliente.

Autenticación

  • CU-01 Registrarse: usuario introduce nombre y contraseña; el sistema crea la cuenta con hash BCrypt y devuelve 201 Created.
  • CU-02 Iniciar sesión: el sistema valida credenciales vía AuthenticationManager y devuelve un token JWT firmado HS256 válido 24 horas.

Gestión de Partida

  • CU-03 Crear nueva partida: el jugador introduce su nombre (hasta 10 caracteres; opciones predefinidas: Jesús, Oro, Ethan) y elige uno de los tres iniciales de Johto (Chikorita 152, Cyndaquil 155, Totodile 158).
  • CU-04 Continuar partida: el sistema carga el estado desde BD (GET /api/v1/juego/estado) y, si no hay red, desde localStorage (pokemon_bytes_save_v1).
  • CU-05 Guardar partida: persiste posición, mapa y JSON de estado cliente en MySQL (POST /api/v1/juego/guardar), y también en localStorage como copia local.
  • CU-06 Reiniciar partida: borra equipo, inventario y PP en BD; restablece dinero a 300 y posición a player-room (5,7) (POST /api/v1/juego/reiniciar).

Exploración

  • CU-07 Moverse por el mapa: jugador se mueve tile a tile con WASD/Flechas; colisiones verificadas antes del movimiento.
  • CU-08 Interactuar con NPCs: activa diálogos estilo GBC con nombre del hablante (Prof. Elm, Mamá, Ayudante, Rival, Aldeano).
  • CU-09 Cambiar de mapa: los warps Tiled disparan WarpSystem con transición fade.

Combate y Captura

  • CU-10 Encuentro salvaje: al caminar sobre hierba, SistemaEncuentros determina si hay encuentro según tabla de apariciones del mapa.
  • CU-11 Ejecutar turno: el jugador selecciona un movimiento; el servidor calcula daño, estados y PP (POST /api/v1/batalla/turno).
  • CU-12 Capturar Pokémon: el jugador lanza una Ball; el servidor ejecuta el algoritmo Gen II.
  • CU-13 Ver equipo y mochila: menú in-game muestra stats de cada Pokémon y agrupación de ítems.

Economía

  • CU-14 Comprar ítems: transacción ACID que descuenta dinero y añade ítem al inventario (POST /api/v1/tienda/comprar).
  • CU-15 Añadir ítems por evento: recompensas del juego (p.ej. poción del PC) añadidas por el servidor (POST /api/v1/juego/inventario/anadir).

6. Arquitectura del Sistema

6.1 Visión General

Pokémon Bytes sigue una arquitectura cliente-servidor con separación clara de responsabilidades. El cliente es una SPA que corre en el navegador; el servidor expone una API REST stateless en el puerto 8081. Ambos se comunican mediante JWT en el header Authorization: Bearer <token>.

6.2 Capas del Backend

  • Controller: recibe peticiones HTTP, delega en servicios. Sin lógica de negocio.
  • Service: lógica de negocio (combate, captura, economía). Usa @Transactional.
  • Repository: interfaces JPA que abstraen el acceso a MySQL.
  • Model/Entity: clases Java anotadas con JPA que mapean las tablas.
  • Security: FiltroAutenticacionJwt valida el token en cada petición y establece el SecurityContext.
  • Component: CommandLineRunner para seeding inicial (Pokédex, movimientos, ítems, usuario-salvajes).

8.2 Seguridad JWT y CORS

La clase ConfiguracionSeguridad define dos SecurityFilterChain con anotación @Order:

  • cadenaActuator (@Order 1): aplica solo a /actuator/**; permite todas las peticiones sin JWT.
  • cadenaFiltroSeguridad (@Order 2): aplica al resto. Deshabilita CSRF (API stateless sin cookies), configura CORS y añade FiltroAutenticacionJwt antes del filtro de login.

El CORS permite orígenes: localhost:*, 127.0.0.1:*, 192.168.*:*, 10.*:* y 172.*:*. Métodos permitidos: GET, POST, PUT, PATCH, DELETE, OPTIONS. Credenciales: true.

ServicioJwt genera tokens firmados con HMAC-SHA256, válidos 24 horas. La clave secreta se configura con la propiedad jwt.secret.key (variable de entorno JWT_SECRET en producción). El método esTokenValido comprueba tanto la firma como la expiración.

11. Implementación – Mecánicas de Juego

11.1 Sistema de Combate Gen II (CalculoService)

Todas las fórmulas matemáticas del motor de combate están encapsuladas en CalculoService como métodos puros sin efectos secundarios, lo que permite su testing unitario aislado.

Fórmula de daño

Daño = ((0.2×N + 1) × A × P / (D × 25) + 2) × STAB × Efectividad × Variación

  N           = Nivel del atacante
  A           = Stat de ataque (reducido al 50% si quemado y movimiento físico)
  P           = Potencia base del movimiento
  D           = Stat de defensa del defensor
  STAB        = 1.5 si tipo movimiento coincide con tipo del atacante, 1.0 si no
  Efectividad = 0.0 / 0.5 / 1.0 / 2.0 / 4.0 (según tabla TIPOS en BD)
  Variación   = random [0.85, 1.0) via ThreadLocalRandom

Resultado mínimo garantizado: 1 (Math.max(1, …))

Golpe crítico

Probabilidad base del 6.25% (random() < 0.0625). Duplica el daño (×2.0).

Verificación de impacto

Si precisionBase ≥ 100 el golpe nunca falla. Si no, random(1,101) ≤ precisionBase.

Cálculo de experiencia

XP ganada = Math.floor(xpBase × nivelDerrotado / 7.0)

Estados alterados – efectos en combate

EstadoEfecto pre-turnoEfecto post-turno
PARALIZADO25% de probabilidad de no atacar.—
DORMIDONo puede atacar; turnosSueno-- cada turno. Se despierta al llegar a 0.—
CONGELADONo puede atacar. 20% de descongelarse.—
QUEMADOAtaque físico reducido al 50%.Daño = hpMax / 8.
ENVENENADO—Daño = contadorToxico × hpMax / 16 (tóxico progresivo).
CONFUSIÓNPosibilidad de golpearse a sí mismo.turnosConfusion--
DRENADORAS—El rival recupera HP a costa del afectado.

11.2 Sistema de Captura (CalculoService.calcularCaptura)

a = ((3 × HPmax − 2 × HPactual) × ratioCaptura × multiplicadorBall) / (3 × HPmax) × bonusEstado

Bonus estado:
  DORMIDO / CONGELADO          → × 2.0
  PARALIZADO / QUEMADO / ENVENENADO → × 1.5
  SALUDABLE                    → × 1.0

Multiplicadores de Ball:
  Poké Ball   → 1.0   |   Súper Ball → 1.5
  Ultra Ball  → 2.0   |   Master Ball → CAPTURE_MAX (garantizada)

Si a ≥ 255 → captura garantizada.
Si no     → ThreadLocalRandom.nextInt(0, 256) < a → captura exitosa.

Los Pokémon capturados se asignan al jugador en POKEMON_USUARIO. Si el equipo está lleno (6 miembros), van a la caja (posicionEquipo ≥ 6).

11.3 Economía e Ítems de Tienda

El jugador comienza con 300 Pokédólares (restablecidos a 300 al reiniciar). La compra de ítems es una operación transaccional ACID anotada con @Transactional: si el saldo es insuficiente, la excepción ErrorNegocio revierte la operación sin modificar el inventario ni el dinero.

El inventario usa la entidad de unión InventarioUsuario con clave compuesta @EmbeddedId InventarioId(usuarioId, itemId) y campo cantidad. Al comprar un ítem ya poseído, se incrementa la cantidad; si es nuevo, se crea el registro. En el frontend, el store normaliza el inventario fusionando filas con el mismo nombre/id para evitar duplicados al sincronizar servidor + caché local.

11.4 Sistema de Guardado Híbrido

El guardado es redundante en tres niveles para garantizar continuidad ante fallos de red:

2. localStorage (cliente) Nombre, mapa, posición, equipo, inventario, flags narrativos, dinero, badges, reloj, esNuevaPartida. Clave: pokemon_bytes_save_v1 (versión 1). usarJuegoStore.guardarPartidaLocal() llamado automáticamente tras guardar en servidor.

13. Diario de Desarrollo

Fase I – Seguridad y Autenticación JWT

Dificultad: Spring Security 6.x eliminó WebSecurityConfigurerAdapter, lo que requirió aprender el nuevo modelo basado en beans de SecurityFilterChain. La documentación oficial mezcla ejemplos de versiones anteriores, causando confusión inicial.

Decisión técnica: Se implementaron dos SecurityFilterChain con @Order: uno exclusivo para Actuator (sin JWT) y otro principal. El filtro FiltroAutenticacionJwt se añade antes de UsernamePasswordAuthenticationFilter. La entidad Usuario implementa directamente UserDetails para simplificar la integración, aceptando que si el sistema de roles crece habrá que extraer un UsuarioPrincipal.

Fase II – Motor de Combate Gen II

Dificultad: Implementar fidedignamente la fórmula de daño de Gen II con todos sus modificadores (STAB, efectividad de tipos, golpe crítico 6.25%, variación [0.85,1.0), penalización por quemadura en físicos) requirió consultar exhaustivamente la Bulbapedia.

Decisión técnica: Se separó toda la matemática pura en CalculoService (métodos sin efectos secundarios) y la lógica de estado en BatallaService. Esto permitió escribir tests unitarios de las fórmulas sin necesidad de Spring ni base de datos. La efectividad de tipos se consultó en base de datos (tabla TIPOS) en lugar de hardcodearla, usando TipoService como intermediario.

Optimización: BatallaService mantiene un ConcurrentHashMap como caché de learnsets (movimientos por especie) para no consultar PokéAPI en cada turno. La versión objetivo es gold-silver para obtener los movimientos correctos de Gen II.

Fase III – Economía e Inventario ACID

Dificultad: Modelar la relación M:N entre Usuario e Item con un campo de cantidad no es posible con colecciones JPA estándar; requirió una entidad de unión explícita con @EmbeddedId y clase @Embeddable.

Decisión técnica: Se creó InventarioUsuario con InventarioId como clave compuesta. La operación de compra está anotada con @Transactional: si el saldo es insuficiente, ErrorNegocio revierte toda la transacción automáticamente.

Fase IV – Algoritmo de Captura

Dificultad: El algoritmo de captura de Gen II usa raíces cuadradas anidadas y podía desbordar int en operaciones intermedias.

Decisión técnica: Se usaron tipos double en todo el cálculo para evitar pérdidas de precisión. El algoritmo simplificado en este proyecto usa una única comparación aleatoria en lugar de las cuatro del juego original, manteniendo la probabilidad equivalente.

Fase V – Data Seeding Reactivo (PokéAPI)

Dificultad: Cargar 251 Pokémon y sus movimientos en el arranque bloqueaba el hilo principal durante varios minutos y sobrepassaba el rate limit de PokéAPI.

Decisión técnica: Se usó Spring WebFlux (WebClient) con Flux.flatMap(..., 5) para limitar la concurrencia a 5 peticiones simultáneas, y .buffer(20) para guardar en batches de 20 en lugar de uno a uno. El seeding se salta si la tabla ya tiene el número esperado de registros.

Integración React + Phaser 3

Dificultad principal del frontend: Phaser gestiona su propio bucle de juego y canvas de forma completamente independiente de React. Zustand pertenece al mundo React; Phaser no tiene acceso directo a los hooks ni al store.

Decisión técnica: Se diseñó PuenteApi.js como módulo singleton que accede al store Zustand mediante usarJuegoStore.getState() (API imperativa, sin hooks). Phaser llama a PuenteApi para todas las operaciones HTTP, y PuenteApi actualiza el store Zustand tras cada respuesta. React escucha los cambios del store de forma reactiva.

Sistema de Mapas Tiled y Colisiones por Grid

Dificultad: El sistema de colisiones de Phaser con física continua causaba que el jugador atravesara paredes en esquinas y que la cámara se desincronizara al cambiar de mapa. Los tilemaps exportados desde Tiled requerían configuración precisa de capas.

Decisión técnica: Se abandonó la física continua en favor de un sistema de movimiento por grid (tile a tile): el jugador comprueba si el tile destino es colisionable antes de iniciar el movimiento, y solo se desplaza si está libre. Esto eliminó todos los bugs de colisión y simplificó enormemente la detección de warps y eventos del mapa.

Sistema de Guardado Híbrido y Consistencia

Dificultad: Al sincronizar el estado del servidor con el localStorage, las versiones antiguas del JSON del cliente mezclaban inventario del JSON con inventario de la BD, duplicando ítems en el store.

Decisión técnica: Se estableció una regla clara: el inventario y el dinero del servidor nunca se sobreescriben con los del estadoClienteJson. La función normalizarListaInventario fusiona filas con el mismo itemId o nombre antes de guardar en el store.

Cinemática de Apertura y Teclado Virtual

Dificultad: Implementar un teclado virtual completo para introducir el nombre del jugador (estilo Game Boy Color) dentro de React, con navegación por teclas de dirección, borrado y confirmación, sin romper el foco del canvas de Phaser.

Decisión técnica: La cinemática de apertura (EscenaApertura.jsx) es un componente React superpuesto al canvas de Phaser. Se implementó un teclado virtual con 26 letras + DEL y confirmación. Las opciones de nombre predefinidas son: "Nuevo nombre", "Jesús", "Oro" y "Ethan". Framer Motion gestiona las transiciones entre el sprite de Oak, Marill y el teclado de nombre.

Estado actual del proyecto: El backend está completamente funcional y probado. El frontend tiene el overworld navegable con 5 mapas, diálogos, la secuencia de apertura con Oak, selección de starter, sistema de encuentros, combate por turnos y menú in-game (equipo, mochila, opciones, guardado). El flujo de retorno del combate al overworld está en refinamiento.

14. Conclusiones

14.1 Lecciones Aprendidas

La separación de responsabilidades salva el proyecto. La arquitectura en capas del backend (Controller → Service → Repository) demostró su valor en múltiples ocasiones: cambiar la lógica de captura no requirió tocar ningún controlador, y añadir el endpoint de reinicio fue trivial porque JuegoService ya tenía los métodos necesarios.

Las pruebas unitarias no son opcionales. Los tests de CalculoService detectaron un error en la fórmula de daño durante la refactorización de la penalización por quemadura. Sin ellos, el bug habría llegado a producción y el combate habría sido incorrecto para todos los Pokémon quemados.

Phaser y React son complementarios, no competidores. La clave fue entender que Phaser es una librería de canvas, no un framework de UI. React gestiona los formularios, menús y overlays donde el DOM es superior; Phaser gestiona el canvas donde el bucle de juego y el renderizado son superiores. El módulo PuenteApi es el pegamento entre ambos.

El seeding reactivo fue imprescindible. Sin WebFlux y la limitación de concurrencia a 5 peticiones, el servidor habría tardado más de 10 minutos en arrancar y habría superado el rate limit de PokéAPI. El enfoque reactivo redujo el tiempo a menos de 2 minutos con batches de guardado.

Definir la fuente de verdad desde el principio. El mayor dolor de cabeza de persistencia se habría evitado si desde el inicio se hubiera documentado que "el servidor es la fuente de verdad para dinero e inventario, y el JSON del cliente nunca los sobreescribe". Llegar a esa regla tardó varias iteraciones de bugs de duplicación de ítems.
